# StupidClaw 文章规划 v3（知乎 / 掘金 / 公众号）

## 1. 总体策略

- 同一技术主线，三种表达方式：  
  - 掘金：工程实现与代码细节。  
  - 知乎：架构判断与方法论。  
  - 公众号：场景价值与可感知结果。
- 固定节奏：每周 1 期主文 + 1 篇短更（复盘/答疑）。
- 每篇只讲一个核心冲突，避免大杂烩。
- 主视觉与话术统一：`StupidClaw = 大智若愚 + 纯文件系统 + 可验证工程闭环`。

---

## 2. 核心叙事主线（7 篇对应教程期）

1. 为什么要做一个“笨但强”的 Agent（发刊词）
2. 先用 Polling 跑通消息闭环，再谈网络架构
3. 从 Polling 平滑升级到 Webhook（可选）
4. Skills 不是越多越好，关键是按需披露
5. `profile.md` 让 AI 记住你，而不是记住噪声
6. Path Jailing：让 AI 有手但不能越狱
7. Cron 让 Agent 从聊天框走向主动服务

发刊词建议主标题：
- 《大智若愚：站在 pi-mono 的肩膀上，手搓极简纯本地 Agent（StupidClaw）》

发刊词必须讲清四件事：
- 为什么拒绝过度工程（不用数据库、复杂 UI、重框架堆叠）。
- 为什么选 `pi-mono`（成熟、简洁、少状态机 bug）。
- 为什么选 Telegram 作为 Message as UI（先拿到可用闭环）。
- 为什么坚持 `.stupidClaw/` 沙盒（AI 可写数据但不可碰源码）。

## 3. 平台差异化选题模板

### 3.1 掘金（工程导向）

内容结构：
- 问题背景（200 字内）
- 数据结构定义（jsonl/profile/cron）
- 关键实现（核心代码片段）
- 验收步骤（命令 + 预期结果）

标题建议：
- 《先别折腾公网：用 Telegram Polling 在本机跑通 Agent 第 1 期》
- 《从 Polling 升级 Webhook：不改业务层的传输层改造实战》
- 《Path Jailing 实战：如何把 AI 文件权限锁死在 .stupidClaw》
- 《Cron + Skills：让本地 Agent 每天 8 点主动发早报》

转化动作：
- 文末固定放 GitHub 仓库地址。
- 评论区置顶“下一期预告 + 征集问题”。

---

### 3.2 知乎（认知导向）

内容结构：
- 行业现象（为什么现在很多 Agent 项目越做越重）
- 反例拆解（过度工程导致的问题）
- StupidClaw 的约束设计（KISS + 数据边界）
- 可复制结论（给读者可执行 checklist）

标题建议：
- 《为什么 2026 年做 Agent，反而应该回到文件系统？》
- 《很多 Agent 项目死在复杂度：我用 KISS 重做了一遍》
- 《Webhook 不是起点：先本机跑通，再升级公网回调》
- 《AI 可以写文件，但绝不能写源码：Path Jailing 的工程意义》

转化动作：
- 结尾附“完整系列导航”。
- 设置话题问题引导评论（例如“你最不能接受哪种过度抽象？”）。

### 3.3 公众号（场景导向）

内容结构：
- 先给结果截图/GIF（主动早报、记住偏好）
- 再讲最少技术背景
- 最后给“你也能做”的三步操作

标题建议：
- 《不用公网也能跑：我在本机做了一个 Telegram AI 管家》
- 《我做了个不会忘事的 AI 管家：每天早上主动给我发简报》
- 《不靠数据库，AI 也能有长期记忆：一个 profile.md 就够了》
- 《让 AI 在沙盒里干活：它很能干，但碰不到你的源码》

转化动作：
- 文末引导加入读者群或订阅下一期。
- 提供“本期代码清单图”降低阅读门槛。

## 4. 发布排期（建议 4 周）

- 第 1 周：发刊词 + 第 1 期
- 第 2 周：第 2、3 期
- 第 3 周：第 4、5 期
- 第 4 周：第 6、7 期 + 系列总结

短更安排：
- 每周一次“踩坑复盘”（Polling 升级 Webhook/Cron/权限问题）。
- 每周一次“读者问答”（从评论提炼）。

---

## 5. 每篇文章固定骨架（复用模板）

1. 这篇要解决什么问题（1 段）
2. 数据结构怎么定义（1 图或 1 段）
3. 最小实现代码（只给关键链路）
4. 验收清单（3-5 条）
5. 下一篇预告（1 段）

保持这个骨架，你的连载会稳定、可读、可持续。

每篇必须附“项目结构与关键代码”小节（建议 6-12 行）：
- 项目结构：列出本期新增/修改文件。
- 关键代码：只列 2-4 条核心链路，不贴大段无关代码。
- 验收命令：给出最小可执行命令与预期输出。

---

## 6. 爆点与风险控制

爆点：
- `pi-mono` + `MiniMax M2.5` + 纯文件架构。
- Path Jailing（安全感强，讨论度高）。
- Cron 主动推送（结果可视化强）。

风险：
- 文章太长导致读者流失。
- 代码跨度过大导致跟不上。
- 概念过多导致“像宣讲不像教程”。

控制办法：
- 单篇只承载一个核心目标。
- 每篇都给可运行验收步骤。
- 术语不超过 3 个，重复解释同一套数据模型。
- 发刊词不用“万能架构”口吻，必须写出 out-of-scope 清单，防止误导读者。

这套规划追求的是持续产出，不是一次性写爆。先稳，再快。
